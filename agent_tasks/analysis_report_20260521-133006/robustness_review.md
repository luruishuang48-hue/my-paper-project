# 稳健性与风险审查：基于 `proposal.md` 与 `事件集数据.csv`

## 一、审查结论

当前 `事件集数据.csv` 可以支持一份数据诊断型、探索型分析报告，但还不能直接支持强因果、强机制或公司关系异质性的正式结论。最核心的问题是：数据虽然有 5160 行，但结构上是 60 个模型发布事件 × 86 家公司；真正的事件-公司关系字段 `relationship` 完全为空，`potential_us_exposure_type` 只是事件层面的潜在暴露类型，不是每家公司与事件之间的关系编码。因此，任何报告都必须避免把 5160 行当作 5160 个独立识别单元，也不能把每家公司都解释为与每个事件存在真实经济暴露。

若要生成正式实证报告，建议把当前结果定位为“事件集与变量可用性审查 + 初步描述统计 + 后续稳健性方案”，而不是完整的因果检验结果。

## 二、数据结构审查发现

1. `事件集数据.csv` 的实际编码是 `gb18030`，不是 UTF-8。读取时需要显式指定编码，否则中文字段会乱码。

2. CSV 存在双表头结构。第一行是中文字段名，第二行是英文变量名映射，第三行以后才是观测。若直接用第一行作为列名，第二行会被误读为一条数据；若直接跳过中文表头，又会丢失部分中文含义。报告生成前应先构造干净字段名，并记录处理方法。

3. 清理第二行表头后，数据为 5160 条观测、99 个字段。事件数为 60，公司数为 86，每个事件正好对应 86 家公司，没有重复的 `final_event_id` × `company_id` 行。

4. `relationship` 字段 5160 行全为空。也就是说，当前文件没有完成 proposal 要求的 `publisher`、`parent_company`、`strategic_partner`、`major_investor`、`cloud_provider`、`direct_competitor`、`compute_supplier`、`application_exposed_firm`、`weak_related_entity` 等事件-公司关系编码。

5. `potential_us_exposure_type` 字段有值，但它是事件层面的潜在暴露类型，且在同一个事件的 86 家公司中重复。它不能替代公司层面的 `relationship`。

6. 样本并非严格“美股上市公司”集合。粗略按 ticker 格式判断，约 3900 行像美股 ticker，约 1260 行含有 `KS`、`HK`、`GR`、`JP`、`TT`、`LN`、`SW` 等非美股或非美国市场代码。若论文题目强调“美股市场证据”，必须说明这些公司是否用于全球 AI 股票池、ADR 样本、竞争公司池，还是应从主样本剔除。

7. 控制变量缺失较多。`size (log_assets)` 只有 2100 条非缺失，`BM_Ratio` 只有 2089 条非缺失；若直接加入这些控制变量，样本会大幅缩小，并可能引入选择性缺失偏误。

8. CAR 存在极端值。`car_10` 最小值约为 -2.37，`car_15` 最小值约为 -2.77，`car_20` 最小值约为 -3.08；`Momentum` 最大值约为 33.64，`volatility` 最大值约为 7.80。这些值不一定错误，但必须做数据核验、缩尾和影响点诊断。

9. 媒体态度字段命名和含义需要复核。部分列名为 `windows-2`、`windows-3` 等，且同一事件内数值看起来重复，可能是事件层媒体变量，而不是事件-公司层变量。报告不能把它解释为每家公司专属媒体情感，除非另有原始文本匹配证据。

10. 文件中没有明显的 ASVI 字段。proposal 强调 ASVI 与媒体情感机制，但当前 CSV 不能直接支持 ASVI 机制检验。

## 三、必须避免的过度解释

### 1. 研究设计层面

1. 不要声称当前结果识别了“重大 AI 模型发布对美股 AI 相关公司股价的因果影响”。模型发布不是随机事件，事件窗口内可能有同时发生的财报、宏观消息、行业新闻和公司公告。更合适的表述是“短窗口市场反应”“事件窗口相关性”或“异常收益的描述性证据”。

2. 不要把短期 CAR 解释为长期基本面变化。报告只能讨论市场预期、短期重新定价或投资者反应，不能直接推断收入、利润率、市场份额、研发效率或长期竞争优势已经变化。

3. 不要把 5160 行观测当作 5160 个独立事件。真正的事件数只有 60，事件层变量在 86 家公司之间重复。若用普通 OLS 标准误，显著性会被夸大。

4. 不要把“AI 相关股票池”解释为真实经济暴露关系。当前每个事件都连接同一组 86 家公司，不能说明所有公司都受到该事件影响。

5. 不要把非美股样本混入“美股市场证据”的主结论。若保留非美国交易代码，应改称“全球 AI 相关上市公司样本”或在主样本中只保留美股和可交易 ADR。

### 2. 事件研究识别层面

1. 不要把官方发布日期机械等同于资本市场信息日。部分模型存在预告、技术报告、API 开放、媒体报道和 leaderboard 收录多日分布。需要区分官方发布日、首次公开日、主流媒体首次报道日和美股可交易日。

2. 不要忽略非交易日和跨时区问题。Google、OpenAI、Anthropic、中国模型公司和欧洲公司发布信息的时间可能落在美股盘前、盘中、盘后或非交易日。事件日需要按美股交易日重新对齐。

3. 不要把宽窗口 CAR 直接解释为模型发布效应。`car_10`、`car_15`、`car_20` 更容易混入财报、宏观数据、货币政策、行业周期和公司新闻。主报告应优先使用短窗口，长窗口作为稳健性或市场学习的补充。

4. 不要忽视事件重叠。60 个事件中存在同日或近日期发布，且 Google、OpenAI、Anthropic 等发布者多次出现。若不同事件窗口重叠，同一公司 CAR 可能被多个事件重复归因。

5. 不要把事件前异常收益忽略不报。proposal 中要求使用 `CARpre[-10,-2]` 检查预期反应和信息泄露。当前文件有“异常收益[?10,?2]”类似字段，但列名乱码或含义不清，必须确认后才能使用。

### 3. 事件-公司关系编码层面

1. 不要把 `true_model_creator` 等同于 `company_id`。例如 OpenAI 事件不能把 Microsoft 当作发布者；Anthropic 事件不能把 Amazon 或 Google 当作发布者；xAI 事件不能默认把 Tesla 当作发布者。

2. 不要把 `possible_us_exposed_tickers` 当作已经完成的事件-公司映射。该字段只是候选提示，不能替代逐条事件-公司关系证据。

3. 不要在 `relationship` 为空的情况下报告关系异质性结论。当前数据无法支持“战略伙伴更受益”“竞争者受损”“算力供应商上涨”等公司关系层面的实证判断。

4. 不要事后根据 CAR 正负反推关系方向。关系编码必须来自事件日前或事件日可验证的公开信息，例如投资协议、云合作、API 分发、产品集成、芯片采购、官方公告或权威媒体报道。

5. 不要把弱关联实体放入主回归。Tesla 与 xAI、Microsoft 与 OpenAI、Amazon/Google 与 Anthropic、中国模型与美国云厂商、开源模型与闭源竞争者等关系都需要逐案判定；证据弱的公司应放入扩展样本。

### 4. 变量解释层面

1. 不要把 AA 能力指标解释为事件日投资者完全掌握的信息。AA 指标更像事后标准化能力度量，可能包含事件后评测或后续模型可用性信息。

2. 不要把 LLM Intelligence Index、media Elo、图像/视频/语音模型指标直接合并为一个原始能力变量。不同模态量纲不同，应分样本处理，或使用 within-modality z-score / rank percentile。

3. 不要把价格为 0 的模型简单解释为“免费”或“成本为零”。开源权重、API 定价缺失、试用期、托管成本和推理成本不是同一个概念。价格变量需要单独清洗。

4. 不要把 `is_open_weight_or_open_source` 直接解释为完全开源。开放权重、开放 API、开放模型卡、研究发布和 OSI 意义上的开源不同，需要在报告中使用谨慎措辞。

5. 不要把媒体态度解释为公司层情感，除非新闻语料确实按公司和事件匹配。当前字段更可能是事件层或窗口层变量。

6. 不要报告 ASVI 机制结论。当前 CSV 未包含可用 ASVI 变量，最多能说明“后续需要加入 ASVI 数据”。

### 5. 回归设定层面

1. 不要在 `relationship` 为空时估计 proposal 中的完整关系交互模型。当前可做的最多是事件特征与股票池 CAR 的探索性相关分析。

2. 不要同时加入过多事件层变量、时间固定效应和发布者固定效应。事件数只有 60，事件层解释变量自由度有限，容易过拟合或完全共线。

3. 不要使用普通异方差稳健标准误作为主结果。至少应按事件聚类；由于事件数只有 60，还应报告 wild cluster bootstrap 或随机化推断。若同一公司跨事件重复出现，双向聚类也应作为稳健性。

4. 不要把每个事件的 86 家公司视为相互独立。市场、行业和 AI 主题冲击会造成横截面相关。需要考虑事件层聚类、行业聚类或公司聚类。

5. 不要把缺失控制变量后的样本回归与全样本描述统计混在一起解释。应分别报告全样本、控制变量完整样本、美股样本、高置信关系样本的样本量和特征差异。

6. 不要在同一张表中混合 LLM 能力样本与 media 能力样本后解释统一能力系数。能力机制检验应分 LLM、media 和跨模态标准化三种口径。

### 6. 报告表述层面

1. 避免使用“证明”“导致”“显著提升公司价值”“资本市场确认技术领先”等强表述。建议使用“显示出短窗口相关性”“与异常收益相关”“市场可能重新定价”“结果与某机制相一致”。

2. 避免把不显著结果解释为“市场没有反应”。不显著也可能来自事件日错位、关系编码缺失、事件重叠、样本异质、媒体覆盖不足或标准误较大。

3. 避免把正向平均 CAR 解释为所有公司受益。AI 模型发布可能同时利好云服务、算力供应商和合作方，也可能压制竞争者或应用层公司。平均效应可能掩盖方向相反的暴露。

4. 避免把中国模型、开源模型或低价模型的市场反应简单解释为单一路径。它们可能同时带来成本下降、闭源溢价压缩、应用扩散、算力需求变化和地缘政治风险。

## 四、建议的稳健性与补救方案

### 1. 数据清洗稳健性

1. 明确读取规则：使用 `gb18030` 编码，第二行作为英文变量名，第三行起为观测。

2. 建立字段字典：保留中文原字段、英文变量名、变量层级（事件层、公司层、事件-公司层、窗口层）、是否可用于主回归。

3. 将 `company_id` 分为美股 ticker、ADR、非美国交易代码和无法识别代码。主样本只保留美股/ADR，非美股放入扩展样本。

4. 对 CAR、Momentum、volatility、价格变量做极端值核验，并报告 1%/99% 缩尾、winsorize 前后结果和删除异常点后的结果。

5. 单独报告变量缺失表，尤其是规模、BM、波动率、动量、能力指标和媒体字段。

### 2. 事件样本稳健性

1. 构造主样本：只保留重大模型发布事件，剔除产品集成、agent 产品、搜索/办公功能、纯研究系统和无法确认模型对象的事件。

2. 构造高置信日期样本：只保留 `release_date_confidence` 为官方或人工确认来源的事件。

3. 构造高媒体覆盖样本：只保留至少两家主流财经或科技媒体报道的事件。

4. 对同日或同窗口多事件做合并、剔除或重叠窗口标记，避免重复归因。

5. 使用替代事件日：官方发布日、主流媒体首次报道日、美股下一交易日、API 开放日或模型卡发布日。

### 3. 事件-公司关系稳健性

1. 补齐 `relationship`、`exposure_strength`、`exposure_source` 和 `manual_exposure_confidence`。没有这些字段前，不应报告关系异质性主结果。

2. 主样本只保留高置信关系，例如发布者、母公司、明确投资者、云合作方、分发方、算力供应商和直接竞争者。

3. 弱关联、AI basket 成员和概念股只用于扩展分析，不进入主结果。

4. 对典型高风险案例做人工复核：OpenAI-Microsoft、Anthropic-Amazon/Google、xAI-Tesla、DeepSeek-NVIDIA/云服务商、Meta 开源模型与闭源竞争者、Google media 模型与 Adobe 等创意软件公司。

5. 使用 leave-one-relationship-out 稳健性，检查结果是否由某一类关系驱动。

### 4. 事件研究稳健性

1. 主窗口建议使用 `car_1` 或明确对应 \,[-1,+1\,] 的字段；`car_3`、`car_5` 作为稳健性；`car_10` 以上窗口只作扩展，不作主识别。

2. 重新计算 CAR 时使用市场模型、Fama-French 三因子、Fama-French 五因子和行业调整收益四种口径。

3. 加入事件前 CAR 或 pre-trend 检查，剔除事件前已显著反应的事件-公司对。

4. 剔除事件窗口内有财报、并购、诉讼、监管、重大产品发布、宏观数据和行业政策冲击的事件-公司对。

5. 对事件日错位做敏感性分析：T0、T0+1、首次媒体报道日和美股下一交易日。

### 5. 回归稳健性

1. 把当前数据的回归定位为探索性。正式回归必须在关系编码完成后进行。

2. 主标准误按事件聚类；补充公司聚类、事件 × 公司双向聚类和 wild cluster bootstrap。

3. 对事件层变量的显著性，报告以事件为单位聚合后的回归或 two-step 回归，避免 86 家公司重复放大事件层自由度。

4. 控制固定效应时保持简洁：行业固定效应、季度或年度固定效应可以作为主设定；发布者固定效应、事件固定效应与事件层能力变量可能共线，应谨慎使用。

5. 分样本估计 LLM、media、中国发布者、美国发布者、开源/闭源、Tier 1/Tier 2，而不是在小样本中堆叠过多交互项。

6. 对连续变量使用标准化、缩尾和 rank transformation 三种口径。

7. 对能力变量使用三种替代指标：AA 原始指标、同模态分位数、事件时点可得 benchmark 或媒体 SOTA 标记。

### 6. 机制检验稳健性

1. ASVI 机制必须等 ASVI 数据加入后再报告。主口径应为 firm-level ASVI，补充 model-level、creator-level 和 event-level attention index。

2. 媒体情感应区分事件级、发布者级、公司级和事件-公司级。当前字段若无法证明公司级匹配，只能作为事件级媒体变量。

3. 媒体情感模型建议至少用 FinBERT、人工校验子样本和替代情感模型进行稳健性比较。

4. 对报道数量和报道情感同时建模，避免把“媒体关注多”误解释为“情感更正面”。

5. 市场学习机制应谨慎表述。时间趋势与 2024 年后阶段划分可以作为描述性证据，但不能证明市场学习的因果过程。

## 五、建议报告写法

建议报告采用以下定位：

1. “本文基于 60 个重大 AI 模型发布事件和 86 家 AI 相关上市公司构造事件-公司面板，初步检验事件窗口异常收益的分布特征。”

2. “由于当前数据中的事件-公司关系编码尚未完成，关系异质性和机制回归应作为后续扩展，而非本文当前主结论。”

3. “当前结果应理解为短窗口资本市场反应的描述性证据，不应直接解释为长期基本面影响或强因果效应。”

4. “所有事件层变量的推断均按事件聚类，并在事件数较少的限制下进行保守解释。”

5. “LLM 与 media 模型能力指标分开报告；跨模态比较只使用标准化分位数，不比较原始能力分数。”

## 六、优先修改清单

1. 先修复数据读取和字段字典，确认第二行英文变量名不是观测。

2. 明确主样本是否只保留美股/ADR；若保留非美国交易代码，修改报告题目或单独列为全球扩展样本。

3. 补齐事件-公司关系编码，这是正式实证报告的前置条件。

4. 复核 CAR 窗口定义，确认 `car_1`、`car_2`、`car_3`、`car_5` 分别对应哪个事件窗口。

5. 复核媒体字段含义，区分事件层、公司层和事件-公司层。

6. 加入 ASVI 数据后再做关注度机制检验。

7. 对极端 CAR、Momentum、volatility 和价格变量做核验与缩尾。

8. 使用事件聚类、双向聚类和 wild cluster bootstrap，避免显著性被重复事件-公司展开结构放大。

9. 在正式报告中把“探索性分析”“主样本结果”“稳健性结果”“尚不能支持的机制结论”分开写。
